**Step 2 — Scrub each of the 3 morph zones via arrow keys.**

The 4 stages live at scroll progress windows (per D-02):
- hero hold: `[0, 0.10]` (10% of total scroll)
- wow hold: `[0.20, 0.55]` (35%)
- feature-a hold: `[0.65, 0.78]` (13%)
- feature-b hold: `[0.85, 1.0]` (15%)

The 3 morph zones (where transitions happen):
- **Morph 1 (hero→wow):** `[0.10, 0.20]` — screen scales from 0.55 to 1.00, opacity 0 to 1
- **Morph 2 (wow→feature-a):** `[0.55, 0.65]` — screen scales 1.00 to 0.50, x 0 to -28vw
- **Morph 3 (fA→fB):** `[0.78, 0.85]` — screen x slides -28vw to +28vw, with scale dip to 0.45 at midpoint (~0.815)

For each morph zone, scrub slowly. The page is `h-[400lvh]` so 100% scroll = 4× viewport.

**Step 3 — At each 25/50/75 of each morph, judge the midstate.**

Sign-off questions for each morph:
- Is there ANY broken-looking moment (layout collapses, image clipped, chrome misaligned)?
- Does the chrome (Mac dots + URL bar) stay attached to the screenshot at every progress?
- Does the box-shadow stay continuous (no seams or duplicates)?

**Specific midstate to verify (Morph 3 — fA→fB cross-center):**

**Step 4 — Verify tonal contrast at each stage:**
Pause at progress 0.05 (hero hold), 0.35 (wow hold), 0.71 (fA hold), 0.92 (fB hold). At every position, the screenshot should retain photorealistic styling (sharp edges, real shadow, white URL bar) — NOT flattened to match the paper-card pencil texture.

**Step 2 — Open Chrome DevTools → Network panel.**

Filter by "Img". Reload the page (Cmd+Shift+R for hard refresh, no cache).

Expected pattern:

- The Initiator column shows `(index)` or `<link>` (NOT `<img>` — that would mean the preload didn't fire and the image came in as a regular src request)

**Step 3 — Open Chrome DevTools → Performance Insights tab → "Reload page".**

Wait for the recording to complete. Look at the LCP section.

Expected:

- The `<img>`'s URL ends in `.avif` (NOT `.png` or `.webp`)

If LCP element is the hero video poster (`/hero/teacher-illustration.png`), Phase 2's hero stage is winning the LCP race over Phase 3's product screen — surface as a TUNE finding (may need preload-priority adjustment, but expected behavior is product screen wins because of the explicit fetchPriority="high" preload).

**Step 4 — Verify `<link rel="preload">` is in the document `<head>`:**

In DevTools → Elements panel, navigate to `<head>`. Look for:
```html
<link rel="preload" as="image" type="image/avif"
      imagesrcset="/hero/profiles-screen-640.avif 640w, ..."
      imagesizes="(min-width:1280px) 1280px, 100vw"
      fetchpriority="high">
```

Note: lowercase HTML attributes (`imagesrcset`, `imagesizes`, `fetchpriority`) are CORRECT — React 19 serialized the camelCase JSX props (`imageSrcSet`, etc.) to lowercase HTML. This is the spec.

If the `<link>` is missing entirely, the head() callback didn't run — surface as BLOCK; executor should re-run `pnpm test src/routes/index.head.test.tsx` and check `Route.options.head?.()` introspection.
